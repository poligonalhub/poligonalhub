# Índice de Módulos

Visão Geral Filosofia e Fundamentos Topologia de Diretórios Modelagem por Composição Sistema Narrativo Central Gerenciadores Barramento Central Core Loop (Fluxo de Execução) Léxico de Tags Decisões Arquiteturais (RDAs) Plano de Desenvolvimento

A MABeTe (Motor Arterial Baseado em Texto) é um motor ágil e modular desenvolvido exclusivamente em Blueprints na Unreal Engine 5. Seu propósito estrutural é modernizar o desenvolvimento de aventuras em texto e ficção interativa através do paradigma de Data-Driven Design (Design Orientado a Dados).

A arquitetura da MABeTe atua fundamentalmente como um "leitor de cartuchos" agnóstico. Ela separa de forma estrita o código-fonte da engine dos dados do jogo em si. Essa fundação permite que criadores e roteiristas desenvolvam universos inteiros preenchendo planilhas e instanciando variáveis lógicas, sem precisar tocar na programação central do motor.

Para ilustrar essa separação, podemos observar a relação entre o motor e o seu primeiro projeto, o Gaiat.OS ou o CartuchoDemo. Enquanto a MABeTe opera nos bastidores como a engrenagem matemática invisível que gerencia o estado do mundo, o cartucho inserido nela fornece a camada estética e o conteúdo narrativo de forma totalmente encapsulada.

Zero Atrito Mecânico O fluxo de trabalho do roteirista deve ser limpo. Game Designers não precisam, sob nenhuma hipótese, abrir um script visual (Blueprint) ou programar lógicas para instanciar novas salas, itens ou diálogos. Toda a expansão do mundo ocorre preenchendo Data Assets.
Efeito Matrioska (Encapsulamento) O núcleo do motor (Engine) é isolado do conteúdo do jogo (Cartucho Narrativo). O cartucho hospeda o texto, os personagens e a estética, mas é a MABeTe quem dita a física, valida ações e guarda a memória. Se o cartucho for ejetado, a engine sobrevive em seu estado puro, sem quebrar.
Internacionalização Nativa (i18n) É expressamente proibido chumbar textos no código (hardcoding). A MABeTe e o seu Léxico foram concebidos para apontar dinamicamente para tabelas de strings de localização nativas da Unreal. O jogo nasce pronto para ser traduzido de forma indolor e simultânea.
Barreira Sanitária do Compilador Errar é humano, compilar falhas é inaceitável. Através de overrides no núcleo de validação (IsDataValid), o motor cria uma barreira no próprio editor. Se o Game Designer referenciar uma sala apagada ou uma Gameplay Tag inexistente, a compilação é bloqueada, impedindo o silently fail em runtime.

A estrutura de diretórios divide claramente o núcleo funcional do motor (Core) e a especificação de conteúdo narrativo/geográfico do cenário ativo (Game/Cartucho). Essa separação é vital para a operação isolada da engine e indexação via Asset Manager.

Para evitar redundância de dados fundamentais (como Nome, Descrição e Chaves de Validação), a MABeTe implementa o padrão de Struct Composition (Composição de Estruturas). Em vez de recriar esses campos em salas, saídas e móveis, todos eles contêm internamente uma F_EntityBase.

F_EntityBase

  • +ID: GameplayTag
  • +Name: Text
  • +Description: Text
  • +RequiredTag: GameplayTag
  • +InvertReq: Boolean
  • +LockMessage: Text
  • +SFX: SoftSoundBase

F_Room

  • +BaseData: F_EntityBase
  • +Exits: Array<F_Exit>
  • +Fixtures: Array<F_Fixture>

F_Exit

  • +BaseData: F_EntityBase
  • +TargetRoom: GameplayTag
  • +Direction: GameplayTag

F_Fixture (Itens/Interativos)

  • +BaseData: F_EntityBase
  • +IsPortable: Boolean
  • +DoubleDispatch: Interfaz (Lógica Invertida)

F_Reaction

  • +ResultText: Text
  • +GrantedTag: GameplayTag
  • +TriggerEvent: F_StateUpdate

Essa arquitetura permite que se novos atributos globais precisarem ser adicionados no futuro (como indicador de acessibilidade ou suporte a voz sintetizada), basta alterar a struct raiz F_EntityBase para propagar a mudança automaticamente.

> O Cadeado Universal Homogêneo

Com a composição em F_EntityBase, cada entidade do jogo (seja a porta de uma sala, uma gaveta, ou um NPC) possui inerentemente um "Cadeado Universal" formado pelas propriedades RequiredTag e InvertReq.

A função EvaluateRequirement atua como um validador puro e agnóstico dentro da MABeTe. Antes de qualquer interação, ela consulta o Container Epistêmico (memória do jogo) para verificar se o jogador possui a Tag exigida. Se a Tag for validada, a ação procede. Caso contrário, a LockMessage é devolvida e a ação é barrada — tudo isso sem a necessidade de programar lógicas específicas ou árvores de IF/ELSE para cada porta ou item.

// Exemplo de Bloqueio via Cadeado Universal num F_Exit (Porta de Saída Genérica)
F_Exit PortaEstudio = {
    BaseData.ID: State.Unlocked.RM_Studio,
    BaseData.Name: "Porta de Carvalho",
    BaseData.RequiredTag: Inventory.IT_Key_Studio, // A tag (Chave) exigida para destrancar
    BaseData.InvertReq: false,
    BaseData.LockMessage: "A porta do estúdio está trancada. Você precisa da chave.",
    TargetRoom: RM_Studio
};

O cérebro da MABeTe não reside num ator temporário no mapa, mas sim num subsistema persistente e imortal derivado da GameInstance (BP_AGI - Arterial GameInstance). Atuando como um Service Locator absoluto, ele detém a verdade sobre o tempo, o espaço e as memórias do jogador.

> A Máquina de Estados (E_GameState)

1. Awaiting Input

Estado de repouso absoluto. O motor aguarda o jogador digitar. O uso de CPU é próximo a zero. Áudio e animações correm em idle via GPU.

2. Processing

O Parser entra em ação. Analisa a string de entrada, filtra stopwords, busca sinónimos no Léxico e tokeniza a ação. A UI fica bloqueada contra spam de inputs.

3. Executing

A física do jogo. O Event Bus é acionado, disparando reações em cadeia, resolvendo validações, imprimindo o output textual e reproduzindo os gatilhos sonoros.

4. Loading

Estado de suspensão volátil para descarregamento de Garbage Collection (GC) assíncrono durante a transição de salas, mascarado diegeticamente na UI.

> O Container Epistémico (A Memória)

Diferente de jogos tradicionais que separam fisicamente o inventário das variáveis de missão, a MABeTe unifica todas as posses, chaves e conhecimentos adquiridos pelo jogador num gigantesco cofre de dados chamado Container Epistémico.

Natureza da Tag no Container Registo em Memória Como o motor interpreta?
Inventário Físico Inventory.IT_Key_Rusty "O jogador possui a Chave Enferrujada na sua posse."
Fato Narrativo (Conhecimento) State.Story.KnowsAboutMurder "O jogador descobriu sobre o assassinato." (Libera novos nós de diálogo nos NPCs).
Alteração de Mundo (Permanência) State.Unlocked.RM_Kitchen "A porta da cozinha foi arrombada e deve continuar aberta ao revisitar a sala."

> Buffer de Contexto Vocabular

A naturalidade da ficção interativa exige que o motor se lembre do que estava a ser discutido. O BP_AGI mantém atualizada a variável LastExaminedTarget, permitindo a resolução de pronomes ("ele", "ela", "isso") no turno subsequente.

Input Jogador: "examinar porta de ferro"

[System] LastExaminedTarget modificado de (Null) para (IT_Door_Iron)


Input Jogador: "abrir ela"

[Parser] Pronome detetado. Substituindo 'ela' pelo Buffer -> "abrir IT_Door_Iron"

[Parser] Evaluator validou a ação. A disparar Evento de Abertura...

Para contornar a limitação de Subsystems nativos para algumas lógicas locais e abraçar uma topologia orientada a componentes, a lógica ativa do turno roda em 6 Managers. Eles são instanciados e inicializados no BeginPlay do BP_PlayerController, seguindo o padrão de Dependency Injection.

BPC_NavigationManager

Responsável pela movimentação espacial e fluxo entre as salas do mundo textual, resolvendo caminhos e verificando cadeados dinâmicos nas saídas.

[Lógica de Resolução]
1. Recebe TargetRoom: RM_Hall
2. Checa o Cadeado na F_Exit local.
3. Sucesso: Muda a variável global SalaAtiva e solicita a Descrição da nova sala.

BPC_InventoryManager

Gerencia a bolsa do jogador, regras de posse, verificação de requerimentos e acumulação de Gameplay Tags associadas a itens guardados.

BPC_Parser

O tradutor neural. Faz o processamento sintático do input do jogador em comandos de máquina utilizando o Data Asset unificado PDA_Lexic.

[Pipeline de Tokenização]
Input: "Pegar a chave enferrujada"
// 1. Remove stopwords ("a")
String Limpa: "Pegar chave enferrujada"
// 2. Busca sinónimos no Léxico
Output: Action.Take + IT_Key_Rusty

BPC_DialogueManager

Gestão ativa de grafos narrativos, nós de conversação ramificados e controle de estado em conversas com NPCs.

BPC_AudioManager

Orquestrador reativo. Escuta atualizações de estado do sistema de eventos e executa crossfades de música atmosférica ou SFXs locais.

BPC_StateManager

A ponte de barramento. Ouve os componentes de input, formata o F_StateUpdate e despacha o evento para o BP_AGI e módulos da UI.

Nenhum Manager conversa de forma hardcoded com sistemas reativos (UI, Áudio, etc). A MABeTe institui uma comunicação totalmente desacoplada através de um sistema de Event Bus.

Em vez de chamar funções diretas, utilizamos a estrutura global F_StateUpdate. O BPC_StateManager expõe a função de interface PublishEvent(F_StateUpdate), que encapsula os seguintes dados:

  • Payload Event Tag: (Gameplay Tag) Define o que aconteceu (ex: Event.Item.Collected).
  • Instigator: (Actor opcional) Quem causou o evento no fluxo atual.
  • Target: (Object opcional) O alvo do evento (o Item, a Sala ou NPC afetado).
  • Magnitude: (Float) Valor numérico auxiliar (ex: quantidade de itens, nível de alteração).

Sempre que ocorre uma mudança sistêmica, o PublishEvent é chamado. O BPC_StateManager processa a informação, atualiza o Container Epistêmico e realiza o broadcast para a UI e o Áudio sem acoplamento direto.

Para materializar a filosofia de Data-Driven Design e separar a lógica da narrativa, a MABeTe obedece a um ciclo de vida estrito e unidirecional a cada interação do jogador. O fluxo nunca deve ser contornado.

[01. Input & Interação]

O jogador digita um comando no Terminal e pressiona a tecla Enter. A Máquina de Estados transita de Awaiting Input para Processing. Imediatamente, a UI congela o campo de texto para impedir o envio de comandos em spam que poderiam corromper o barramento.

[02. Scanner / Limpeza]

A string crua viaja para o BPC_Parser. Caracteres especiais, acentuações e pontuações são removidos. Palavras vazias (stopwords como "o", "a", "de", "para") são sumariamente descartadas da matriz de análise.

[03. Parser Neural & Léxico]

A string limpa é cruzada com o Data Asset PDA_Lexic. O motor identifica sinônimos e tenta isolar uma Ação e um Alvo. Se o jogador usar pronomes ("pegar ela"), o Parser consulta o Buffer de Contexto Vocabular para resgatar o último objeto examinado. A linguagem humana é convertida num par algébrico seguro: Action.Take + IT_Key.

[04. Evaluator (O Tribunal)]

Antes da execução, a MABeTe submete a intenção à função EvaluateRequirement. O Cadeado Universal do alvo é inspecionado. Se o jogador não tiver as Tags exigidas (ex: falta a chave para abrir a porta), o fluxo é abortado e a LockMessage predefinida pelo roteirista é preparada para o log.

[05. Publish Event (Mutação)]

Se o Cadeado for validado com sucesso, o BPC_StateManager aciona o PublishEvent. A Gameplay Tag do Evento transita pela arquitetura e o Container Epistêmico (Memória de Fatos) é atualizado, persistindo a mudança no mundo de forma perene.

[06. Output Visual & Auditivo]

O Barramento notifica os componentes periféricos. O Áudio reage com a reprodução de um SFX. A UI acorda, imprime o texto da reação (ResultText) utilizando o efeito de Typewriter e desbloqueia o teclado. O sistema retorna pacificamente para o estado Awaiting Input.

Como definido na decisão RDA-002, todo o fluxo narrativo e mecânico é mapeado usando a hierarquia rigorosa de Gameplay Tags do motor.

Para mitigar o risco de colisões de nomenclatura no Content Browser e garantir determinismo absoluto, a realidade MK2 introduz uma gramática estrita para a criação de tokens e nomenclatura de ficheiros:

[DIRETRIZ DE INTEGRIDADE]

• Fórmula de Mutação de Estado: State.<Verbo/Acao>.<Alvo>

• Prefixos Obrigatórios de Ativos:

- RM_ : Ativos de Ambiente/Salas (Ex: RM_Kitchen)

- IT_ : Todos os Itens e Móveis herdeiros polimórficos de Fixture (Ex: IT_RustyKey)

- STR_ : Chaves puras de localização e strings desacopladas (i18n)

Para garantir a escalabilidade e evitar Dívida Técnica, toda mudança de paradigma na MABeTe é documentada através de RDAs.

RDA-001: GameInstance Subsystem Core Engine

Contexto: Forçar o projeto final a derivar sua classe principal da engine cria uma dependência rígida, limitando a flexibilidade da equipe.
Decisão: Optou-se por isolar todo o núcleo de memória e processamento da MABeTe em um GameInstance Subsystem (AGIS). O motor nasce, opera e morre de forma autônoma e paralela ao ciclo de vida do jogo principal.

RDA-002: Gameplay Tags > Lógicas Booleanas Data Control

Contexto: Rastreamento narrativo baseado em variáveis literais booleanas gera código espaguete e validações frágeis.
Decisão: Abolição total das lógicas booleanas literais para controle de estado. A MABeTe adota exclusivamente o ecossistema hierárquico de Gameplay Tags. O código torna-se limpo e focado na verificação de "Cadeados Universais".

RDA-005: Struct Composition e Delta State Data Modeling

Contexto: Redundância de variáveis espalhadas em dezenas de USTRUCTs independentes dificulta a manutenção em massa.
Decisão: Aplicação do padrão de "Composição de Estruturas" (F_EntityBase). Aliado a isso, exige-se um Delta State Minimalista: é proibido guardar referências a objetos complexos, serializando apenas modificações pontuais de tags.

RDA-010: Editor Utility Widgets (EUW) Tooling

Contexto: Preencher estruturas e criar Primary Data Assets manualmente através do painel nativo da Unreal é lento e sujeito a erros.
Decisão: Desenvolvimento de um painel visual personalizado usando EUW integrado diretamente no editor Unreal, fornecendo campos validados e preenchimento automático.

RDA-011A: Prefixos Estritos e Tokens Data Control

Evitar colisões de nomenclatura e rastrear ativos no editor.
Fórmula inquebrável para estados: State.<Acao>.<Alvo>. Arquivos seguem: RM_ (Salas) e IT_ (Itens/Móveis - Herdeiros polimórficos de Fixture).

RDA-011B: Rigidez Topológica Asset Manager

A descoberta automática via Asset Manager exige caminhos estáticos. Automações em Python foram estritamente proibidas para evitar corrupção externa.
A árvore de diretórios do Cartucho é blindada. O validador bloqueia salvamentos em pastas órfãs no Content Browser.

RDA-013: Validador Silencioso (Sanity Checker) Validation

Contexto: Referências órfãs quebram o motor em runtime silenciosamente.
Decisão: Criação de um script de validação automática. Ao tentar salvar um Data Asset, o validador inspeciona recursivamente os caminhos de PrimaryAssetId. Se estiver corrompido, a compilação é bloqueada.

RDA-014: Rigidez de Topologia Asset Management

Contexto: Organização descentralizada quebra a indexação de PrimaryAssetIds.
Decisão: Blindagem absoluta da árvore de diretórios raiz do Cartucho. O Asset Manager escaneará única e exclusivamente caminhos especificados.

O desenvolvimento e implementação do motor MABeTe é executado através de um cronograma sequencial de entregas críticas (Fases e Hero Cards).
